# 認証・認可標準

[目的: 認証モデル、トークン/セッションのライフサイクル、権限チェック、セキュリティを統一する]

## 方針
- 明確な分離: 認証（誰か） vs 認可（何をしてよいか）
- セキュアバイデフォルト: 最小権限、fail closed、短命トークン
- UX を意識: リスクが高い箇所では摩擦を、それ以外はスムーズに

## 認証

### 方式（選択 + 根拠）
- 選択肢: JWT, Session, OAuth2, ハイブリッド
- 採用: [本プロジェクトの方式]、理由 [理由]

### フロー（概略）
```
1) ユーザーが ID を証明（クレデンシャル or プロバイダ経由）
2) サーバーが検証してトークン/セッションを発行
3) クライアントはリクエスト毎にトークンを送信
4) サーバーがトークンを検証して処理を進める
```

### トークン/セッションのライフサイクル
- 格納: httpOnly Cookie もしくは Authorization ヘッダ
- 期限: アクセストークンは短命、リフレッシュは長め（採用時）
- リフレッシュ: トークンをローテート、失効に従う
- 失効: ログアウト/侵害時にブラックリスト化／ローテート

### セキュリティパターン
- TLS 必須。可能ならトークンを JS から見えなくする
- aud/iss にバインド、クレームは最小限
- 機微操作にはデバイスバインディング、IP/リスクチェック検討

## 認可

### 権限モデル
- いずれかを選択: RBAC / ABAC / オーナーシップ型 / ハイブリッド
- ロール／属性は中央で定義、コード横断でハードコードしない

### チェック（どこで強制するか）
- ルート/ミドルウェア: 粗粒度ゲート
- ドメイン/サービス: 細粒度判定
- UI: 条件付きレンダリング（セキュリティは依存させない）

実装パターン例:
```typescript
requirePermission('resource:action'); // route
if (!user.can('resource:action')) throw ForbiddenError(); // domain
```

### オーナーシップ
- パターン: オーナー OR 特権ロールが操作可能
- 変更前にエンティティ境界で検証

## パスワードと MFA
- パスワード: 強いポリシー、ハッシュ化 (bcrypt/argon2)、平文禁止
- パスワードリセット: 期限付き・1 回限りトークン、ユーザーに通知
- MFA: 高リスク操作で段階的に要求（ポリシー駆動）

## API 間認証
- API Key もしくは OAuth Client Credentials
- スコープは最小に、ローテートし利用を監査する
- ID（ユーザー/キー）単位でレート制限
