# データベース標準

[目的: スキーマ設計、クエリ、マイグレーション、整合性を統一する]

## 方針
- まずドメインをモデリングし、正しさを得てから最適化する
- 制約を明示し、不変条件は DB に強制させる
- 必要なものだけクエリし、最適化前に計測する

## 命名と型
- テーブル: `snake_case`、複数形 (`users`, `order_items`)
- カラム: `snake_case` (`created_at`, `user_id`)
- 外部キー: `{table}_id` で `{table}.id` を参照
- 型: タイムゾーン付きタイムスタンプ、強い ID、金額は精度型

## リレーション
- 1:N: 子テーブルに外部キー
- N:N: 中間テーブル + 複合キー
- 1:1: 外部キー + UNIQUE

## マイグレーション
- マイグレーションは不変。常にロールバックを用意する
- 小さく焦点を絞ったステップ。非本番で先にテストする
- 命名: `{seq}_{action}_{object}`（例: `002_add_email_index`）

## クエリパターン
- 単純な CRUD と安全性は ORM、複雑/性能重要箇所は raw SQL
- N+1 を避ける（eager load / batching）、大きな集合はページングする
- 外部キー・頻繁にフィルタ／ソートするカラムにインデックス

## コネクションとトランザクション
- ワークロードに合わせたプール（サイズ／タイムアウト）を使用
- 作業単位ごとに 1 コネクション。速やかにクローズ／返却する
- 複数ステップの変更はトランザクションで包む

## データ整合性
- NOT NULL / UNIQUE / CHECK / FK 制約を活用する
- 必要に応じて DB 側でもバリデート（多層防御）
- 派生値は生成カラムを優先

## バックアップとリカバリ
- リテンション付き定期バックアップ。リストアをテストする
- RPO/RTO 目標を明文化、バックアップジョブを監視する

---
_パターンと意思決定に集中する。環境固有の設定は別場所。_
