# エラー処理標準

[目的: エラーの分類、形状、伝播、ロギング、モニタリングを統一する]

## 方針
- 可能な限り fail fast、システム境界では段階的縮退
- スタック全体で一貫したエラー形状（人間・機械の両方で読める）
- 既知エラーは発生源近くで処理、未知エラーはグローバルハンドラへ

## 分類（発生源で処理方針を決める）
- クライアント: 入力／バリデーション／ユーザー操作の問題 → 4xx
- サーバー: システム障害／予期せぬ例外 → 5xx
- ビジネス: ルール／状態違反 → 4xx（例: 409）
- 外部: サードパーティ／ネットワーク障害 → 文脈を付けて 5xx か 4xx にマップ

## エラー形状（単一の正規フォーマット）
```json
{
  "error": {
    "code": "ERROR_CODE",
    "message": "Human-readable message",
    "requestId": "trace-id",
    "timestamp": "ISO-8601"
  }
}
```
原則: 安定したコード列挙、シークレットを含めない、トレース情報を含める。

## 伝播（どこで変換するか）
- API レイヤー: ドメインエラー → HTTP ステータス + 正規ボディ
- サービスレイヤー: 型付きビジネスエラーを throw、文字列ベースのエラーを避ける
- データ／外部レイヤー: プロバイダエラーを安全で実行可能なコードでラップ
- 未知エラー: グローバルハンドラへ伝搬 → 500 + 汎用メッセージ

実装パターン例:
```typescript
try { return await useCase(); }
catch (e) {
  if (e instanceof BusinessError) return respondMapped(e);
  logError(e); return respondInternal();
}
```

## ロギング（ノイズより文脈）
記録する: operation, userId（可能なら）, code, message, stack, requestId, 最小文脈。
記録しない: パスワード、トークン、シークレット、フル PII、機微データを含むフルボディ。
レベル: ERROR（失敗）、WARN（回復可能/エッジ）、INFO（主要イベント）、DEBUG（診断）。

## リトライ（安全な場合のみ）
リトライする: ネットワーク／タイムアウト／一時的 5xx かつ操作が冪等。
リトライしない: 4xx、ビジネスエラー、非冪等フロー。
戦略: 指数バックオフ + ジッタ、回数上限、冪等性キーを要求する。

## モニタリングとヘルス
追跡する: コード／カテゴリ別エラー率、レイテンシ、飽和度。スパイク／SLI 違反でアラート。
ヘルス公開: `/health`（live）, `/health/ready`（ready）。エラーをトレースに紐付ける。
