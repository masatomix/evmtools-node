# API 標準

[目的: 命名、構造、認証、バージョニング、エラーで一貫した API パターンを定める]

## 方針
- 予測可能でリソース指向の設計を優先する
- 契約は明示的に、互換性の壊れる変更は最小化する
- セキュアバイデフォルト（認証ファースト、最小権限）

## エンドポイントパターン
```
/{version}/{resource}[/{id}][/{sub-resource}]
```
例:
- `/api/v1/users`
- `/api/v1/users/:id`
- `/api/v1/users/:id/posts`

HTTP メソッド:
- GET（取得、safe、冪等）
- POST（作成）
- PUT/PATCH（更新）
- DELETE（削除、冪等）

## リクエスト/レスポンス

リクエスト（一般形）:
```json
{ "data": { ... }, "metadata": { "requestId": "..." } }
```

成功:
```json
{ "data": { ... }, "meta": { "timestamp": "...", "version": "..." } }
```

エラー:
```json
{ "error": { "code": "ERROR_CODE", "message": "...", "field": "optional" } }
```
（ルールは error-handling を参照）

## ステータスコード（パターン）
- 2xx: 成功（200 取得、201 作成、204 削除）
- 4xx: クライアント問題（400 バリデーション、401/403 認証認可、404 不在）
- 5xx: サーバー問題（500 汎用、503 利用不可）
結果に最も適合するステータスを選ぶ。

## 認証
- クレデンシャルは標準位置に置く
```
Authorization: Bearer {token}
```
- ビジネスロジック前に未認証を拒否する

## バージョニング
- URL / ヘッダ / メディアタイプでバージョン管理
- 破壊的変更 → 新バージョン
- 互換変更 → 同一バージョン
- 廃止予告期間と告知を用意する

## ページング/フィルタ（該当時）
- ページング: `page`, `pageSize` もしくはカーソル方式
- フィルタ: 明示的なクエリパラメータ
- ソート: `sort=field:asc|desc`
ページング情報は `meta` に含める。

---
_パターンと意思決定に集中する。エンドポイントカタログは作らない。_
