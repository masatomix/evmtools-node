# セキュリティ標準

[目的: バリデーション、認可、シークレット、データ取扱いのパターンでセキュリティ姿勢を定義する]

## 方針
- 多層防御、最小権限、セキュアバイデフォルト、fail closed
- 境界でバリデート、出力先に応じてサニタイズ、入力を信用しない
- 認証（誰か）と認可（何をしてよいか）を分離する

## 入力と出力
- API 境界と UI フォームでバリデート、型と制約を強制する
- 出力先に応じてサニタイズ／エスケープ（HTML, SQL, シェル, ログ）
- ブロックリストよりも許可リストを優先、早期に最小情報で拒否する

## 認証・認可
- 認証: ID を検証し、短命トークン／セッションを発行する
- 認可: 操作前に権限を確認、デフォルト deny
- ポリシーは中央化、コード横断で重複チェックしない

パターン:
```typescript
if (!user.hasPermission('resource:action')) throw ForbiddenError();
```

## シークレットと設定
- シークレットを絶対に commit しない。シークレットマネージャか環境変数に置く
- 定期ローテーション、アクセス監査、スコープ最小化
- 起動時に必須環境変数を検証、欠落なら fail fast

## 機微データ
- 収集を最小化、ログでマスク／伏字、保存・転送時に暗号化
- ロール／必要性に応じてアクセス制限、機微レコードへのアクセスを追跡

## セッション/トークンセキュリティ
- 可能なら httpOnly + secure Cookie、すべてに TLS
- 期限短め、リフレッシュ時にローテート、ログアウト/侵害時に失効
- aud/iss バインド、クレームは最小限

## ロギング（セキュリティ視点）
- 認証試行、権限拒否、機微操作を記録
- パスワード、トークン、シークレット、フル PII、フルボディを記録しない
- requestId とコンテキストで相関できるようにする

## ヘッダーとトランスポート
- TLS 強制、HSTS
- セキュリティヘッダー設定（CSP, X-Frame-Options, X-Content-Type-Options）
- モダンな暗号を選択、弱いプロトコル／暗号を無効化

## 脆弱性対応
- セキュアなライブラリを選定、依存を最新に保つ
- CI に静的／動的スキャン、追跡と是正
- 共通脆弱性クラスをチームに啓蒙、上記のパターンに落とし込む
