# 設計書テンプレート

---
**目的**: 異なる実装者の間でも実装の一貫性を確保できるだけの詳細度を提供し、解釈のブレを防ぐ。

**進め方**:
- 実装判断に直結する必須セクションを記載する
- 実装ミスの防止に不可欠でない限り、任意セクションは省略する
- 機能の複雑さに合わせて詳細度を調整する
- 長い文章よりも図表を優先する

**注意**: 1000行に近づく場合は機能が複雑すぎる兆候であり、設計の単純化または複数 spec への分割を検討すること。
---

> セクション順は明確さが上がる場合に並び替えても良い（例: 要件トレーサビリティを前方に出す、データモデルをアーキテクチャ近くに置く）。各セクション内では **概要 → 範囲 → 決定事項 → 影響/リスク** の流れを保ち、レビュワーが一貫してスキャンできるようにする。

## 境界コミットメント

この spec が責任を負う境界を具体的に明示する。アーキテクチャ・タスク・後続バリデーションのアンカーとして扱う。

### この spec が担うもの
- この spec が責任を持つ機能・振る舞い
- この spec が保有または権威となるデータ
- この spec が定義・安定化させるインターフェース/契約

### 境界の外
- この spec が明示的に**担わない**関連事項
- 別 spec / 既存サブシステム / 後フェーズに先送りする作業
- 「ついでに」吸収してはならない変更

### 許容する依存関係
- この設計が依存して良い上流システム/spec/コンポーネント
- 利用して良い共有インフラ
- 違反してはならない依存制約

### 再検証トリガー
依存する spec や利用者側で統合の再確認を強制すべき変更種別を列挙する。

- 契約の形状の変更
- データ所有権の変更
- 依存方向の変更
- 起動/実行時前提条件の変更

## アーキテクチャ

> 詳細な調査メモは背景情報として `research.md` を参照する。design.md はレビュワーが自己完結で読めるよう、すべての決定事項と契約をここに記載する。
> 主要な決定事項はテキストで明示し、構造的な詳細は図に持たせる。同じ情報を文章と図で重複させない。
> 後続の補助セクションは、責任境界・依存ルール・統合ポイントが明確になる場合を除き、できる限り軽量に保つ。

### 既存アーキテクチャ分析（該当する場合）

- 現在のアーキテクチャパターンと制約

## ファイル構成計画

この機能のディレクトリ構造とファイル責務をマッピングする。本セクションはタスクの `_Boundary:_` 注釈や実装タスクブリーフを直接駆動する。粒度は機能規模に合わせて調整:

- **小規模機能**: 個別ファイル単位で責務を列挙
- **大規模機能**: ディレクトリ階層 + ドメイン/モジュール単位のパターンを記述し、自明でないファイルのみ個別に列挙

### ディレクトリ構成
```
src/
├── domain-a/              # ドメイン A の責務
│   ├── controller.ts      # エンドポイントハンドラ
│   ├── service.ts         # ビジネスロジック
│   └── types.ts           # ドメイン型
├── domain-b/              # ドメイン B（domain-a と同じパターン）
└── shared/
    └── cross-cutting.ts   # 自明でないため理由を明記
```

> 繰り返し構造はパターンを 1 回だけ説明する（例: "domain-b は domain-a と同じパターン"）。パスから責務が自明でないファイルのみ個別に挙げる。

### 変更ファイル
- `path/to/existing.ts` — 何をなぜ変更するか

> 各ファイルは責務を 1 つに保つ。一緒に変わるファイルはまとめる。繰り返し構造はパターンを 1 回説明するに留め、すべてのファイルを列挙しない。
> 「コンポーネント・インターフェース」セクションと重複しないこと。ここでは物理的なファイル配置に集中する（コンポーネントとのマッピング）。

## システムフロー

非自明なフローの説明に必要な図のみを記載する。Pure Mermaid 記法を用いる。代表的なパターン:
- Sequence（複数当事者の相互作用）
- プロセス / 状態（分岐ロジック、ライフサイクル）
- データ / イベントフロー（パイプライン、非同期メッセージ）

単純な CRUD 変更ならこのセクションごと省略する。
> フロー単位の決定（ゲート条件、リトライなど）は図の後に短く補足する。各ステップの文字起こしはしない。

## 要件トレーサビリティ

## コンポーネント・インターフェース

詳細に入る前に、まずクイック参照表を提示する。

- サマリは表またはコンパクトなリストで良い。表の例:
  | コンポーネント | ドメイン/レイヤー | 目的 | 要件カバレッジ | 主な依存（P0/P1） | 契約 |
  |----------------|--------------------|------|----------------|---------------------|------|
  | ExampleComponent | UI | XYZ を表示する | 1, 2 | GameProvider (P0), MapPanel (P1) | Service, State |
- 新たな境界（ロジックフック、外部統合、永続化など）を導入するコンポーネントだけ詳細ブロックを書く。単純な表示コンポーネントはサマリ行＋短い実装メモで十分。

詳細ブロックはドメインまたはアーキテクチャレイヤーごとにグループ化する。各詳細コンポーネントでは要件 ID を `2.1, 2.3` のように記載する（"要件" の語を付けない）。複数 UI コンポーネントが同じ契約を共有する場合、基底のインターフェース／props 定義を参照し、コードブロックを重複させない。

## データモデル

この機能で変化するデータ領域に絞る。

### ドメインモデル
- 集約とトランザクション境界
- エンティティ、値オブジェクト、ドメインイベント
- ビジネスルールと不変条件
- 関係が複雑な場合は Mermaid 図を任意で

### 論理データモデル

**構造定義**:
- エンティティ間の関連とカーディナリティ

- 自然キーと識別子

**一貫性と整合性**:
- トランザクション境界
- カスケードルール
- 時間軸（バージョニング、監査）

### 物理データモデル

**リレーショナル DB**:
- 型付きテーブル定義
- 主キー／外部キーと制約
- インデックスと性能最適化

**ドキュメントストア**:

## エラー処理

### エラー処理戦略
各エラー種別に対する具体的なハンドリングパターンとリカバリ機構。

### エラーカテゴリと応答
**ユーザーエラー**（4xx）: 入力不正 → フィールド単位バリデーション; 認可エラー → 認証ガイダンス; Not Found → ナビゲーション補助
**システムエラー**（5xx）: インフラ障害 → 段階的縮退; タイムアウト → サーキットブレーカー; リソース枯渇 → レート制限
**ビジネスロジックエラー**（422）: ルール違反 → 条件の説明; 状態競合 → 状態遷移ガイダンス

**プロセスフロー可視化**（複雑なビジネスロジックがある場合）:
ビジネスワークフローを含む複雑なエラーシナリオのみ Mermaid フローチャートを記載する。

### モニタリング
エラー追跡、ロギング、ヘルスチェックの実装方針。

## テスト戦略

### 既定セクション（ドメインに合わせて名称・項目は調整可）
- ユニットテスト: コア機能/モジュールから 3〜5 項目（例: 認証メソッド、サブスクリプションロジック）
- 統合テスト: コンポーネント横断のフローから 3〜5 項目（例: Webhook 処理、通知）
- E2E / UI テスト（該当時）: クリティカルパス 3〜5 項目（例: フォーム、ダッシュボード）
- 性能 / 負荷（該当時）: 3〜4 項目（例: 並行性、高ボリューム処理）
